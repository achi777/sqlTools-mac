# Database Version Compatibility

The DB Tool is developed against **PostgreSQL 16** and **MySQL 8.0**. TASK 42
verified it against **PostgreSQL 13 / 14 / 15** and **MySQL 5.7**, and TASK 43
added **MariaDB** as a first-class engine (verified on **MariaDB 11**) — all on
disposable Docker containers (separate ports/volumes from the dev setup), driving
the full `DbDriver` layer through the headless smoke suite (18–19 feature groups)
against each version's real server.

## Supported versions (summary)

| Engine | Versions verified | Status |
| --- | --- | --- |
| PostgreSQL | **13, 14, 15, 16** | ✅ Fully supported |
| MySQL | **5.7, 8.0** | ✅ Fully supported |
| MariaDB | **11** (10.3+ for sequences) | ✅ Fully supported |
| Oracle | **XE 21c** (Thin 12.1+; Thick for older) | 🟡 Basics supported (advanced staged) |
| SQL Server | **2022** (2016 SP1+ for CREATE OR ALTER) | ✅ Supported (sequences staged) |
| SQLite | 3.x (bundled) | ✅ Fully supported |

> Older PostgreSQL (≤ 12) was not part of this pass. PostgreSQL 13+, MySQL 5.7+,
> and MariaDB 10.3+ are the supported floors.

## Oracle (basics — advanced features staged)

Oracle is a selectable engine using **node-oracledb** with a **driver-mode
selector** in the connection form:

- **Thin** (default) — pure JS, no external libraries, Oracle **12.1+**. Packages
  cleanly (externalized like the other drivers; no native build).
- **Thick** (optional) — needs **Oracle Instant Client** for older Oracle (11g,
  etc.). We **detect** it via `oracledb.initOracleClient()` and, if it's missing,
  show a clear message (`DPI-1047: Cannot locate Oracle Client library` → "install
  Instant Client or switch to Thin") — we never bundle or auto-install it.

**Verified basics (Oracle XE 21c, Thin) — all PASS:** connect, list tables/views
(`ALL_TABLES`/`ALL_VIEWS`), schema-aware catalog + PK (`ALL_TAB_COLUMNS`/
`ALL_CONSTRAINTS`), paginated browse (`OFFSET/FETCH`, stable `ORDER BY` PK),
autocomplete, the three filter modes producing valid Oracle SQL (`:n` binds;
case-insensitive `contains` via `UPPER()`; `%`-escaped; quote-safe; `>=`,
`BETWEEN`, `IN`, nested funnel AND/OR, custom WHERE, `IS NULL` partition), and
**parameterized grid CRUD by PK** (insert with `RETURNING id INTO`, update,
delete).

**Table designer — valid Oracle DDL (TASK 46):** the designer generates correct
Oracle **CREATE/ALTER TABLE** — `NUMBER GENERATED BY DEFAULT AS IDENTITY`
(never `AUTOINCREMENT`), the primary key declared exactly **once**, Oracle types
via the generic→Oracle mapping (`integer`→`NUMBER(10)`, `varchar`→`VARCHAR2(n)`,
`text`→`CLOB`, `boolean`→`NUMBER(1)`, `timestamptz`→`TIMESTAMP WITH TIME ZONE`,
`json`→`CLOB`, `uuid`→`VARCHAR2(36)`, …), Oracle FK rules (`ON DELETE
CASCADE`/`SET NULL` only, no `ON UPDATE`), and Oracle `ALTER` syntax
(`ADD (…)`, `MODIFY (…)`, `DROP COLUMN`, `DROP/ADD PRIMARY KEY`, `ADD CONSTRAINT
… FOREIGN KEY`, `CREATE/DROP INDEX`). Verified end-to-end (create + insert +
alter + designer round-trip) on the real XE container with **no ORA-00907**.

**Visual View Builder (TASK 49):** Oracle table aliases are emitted **without
`AS`** (`FROM … "t1"`, which Oracle requires — `AS` before a table alias causes
ORA-00933), while column aliases keep `AS`. INNER/LEFT/self joins + WHERE +
aggregates build, preview, and save as views. Other engines keep `AS` unchanged.

**Sequences (TASK 48):** Oracle connections show a real **Sequences** node (list
+ details from `ALL_SEQUENCES`, incl. `LAST_NUMBER` with the cache caveat) and
support **create / alter / restart / rename / drop**. `RESTART` uses
`ALTER SEQUENCE … RESTART START WITH n` on Oracle 12.2+ (version-detected), with
a confirmed **DROP + CREATE** fallback on older versions. IDENTITY-backing
**`ISEQ$$` system sequences are flagged read-only** and never offered for
edit/drop.

**Indexes (TASK 51):** Oracle tables show a real **Indexes** node (list from
`ALL_INDEXES`/`ALL_IND_COLUMNS`, with `UNIQUE` and `STATUS` — a non-`VALID`
`UNUSABLE` index shows a status tag) and support **create / edit / rename /
drop** of standalone B-tree indexes. Rename uses `ALTER INDEX … RENAME TO`
(schema-scoped, like PG — not MySQL's `ALTER TABLE … RENAME INDEX`); a
column/unique change is DROP + CREATE. Constraint-backing (PK/UNIQUE) and system
(`SYS_*`, non-`NORMAL`) indexes are **flagged read-only** — never offered for
edit/drop (drop the constraint in the Table Designer instead).

**Triggers (TASK 52):** Oracle tables show a real **Triggers** node (list from
`ALL_TRIGGERS` joined with `ALL_OBJECTS`) showing timing/event, and — unique to
Oracle — the **ENABLED/DISABLED** status and **VALID/INVALID** compile state as
tags. Supported: **create** (PL/SQL template using `:NEW`/`:OLD`, row- or
statement-level, optional `WHEN`), **edit** via `CREATE OR REPLACE TRIGGER` (no
DROP needed), **enable/disable** via `ALTER TRIGGER … ENABLE|DISABLE`, and
**drop** (schema-scoped `DROP TRIGGER "schema"."name"`). The whole
`CREATE TRIGGER` is executed as **one statement** (the PL/SQL body's `END;` is
preserved, not split on `;`). Crucially, because Oracle **creates a trigger even
when its PL/SQL fails to compile** (leaving it `INVALID`), the app checks
`ALL_ERRORS` after applying and **surfaces the compile error (line/col/text)
instead of a false success** — the trigger then shows `INVALID` in the tree.
Definitions load via `DBMS_METADATA.GET_DDL` with an `ALL_TRIGGERS`
reconstruction fallback. Multi-event combos (`INSERT OR UPDATE`) display in the
tree but the editor maps to a single event — full combo editing is backlog.

**Functions, procedures & packages (TASK 53):** the **Functions** and
**Procedures** nodes now list standalone routines (from `ALL_OBJECTS`) with a
signature built from `ALL_ARGUMENTS` and a **VALID/INVALID** status tag — routines
a user created by hand previously did not appear at all. A new Oracle-only
**Packages** node lists PL/SQL packages, marks whether each has a body
(`spec only` tag) and whether either part is `INVALID`, and offers **Edit spec**,
**Edit body**, **Drop body**, and **Drop package** (dropping the package removes
spec + body). Create uses **correct Oracle templates** — `RETURN` (not `RETURNS`),
`IS`/`AS`, `IN`/`OUT` params, and a package **spec + body** pair separated by a
lone `/` line — so objects compile without hand-fixing. Edit loads the source via
`DBMS_METADATA.GET_DDL` (falling back to `ALL_SOURCE`) and applies with
`CREATE OR REPLACE` (no DROP). Each statement runs whole (PL/SQL `END;` preserved,
never split on `;`), and — as with triggers — a routine that **compiles with
errors is reported as a failure with the `ALL_ERRORS` line/text**, not a false
success, and shows `INVALID` in the tree.

**SQL export & DB dump (TASK 55):** exporting an Oracle table to `.sql` (and the
whole-database dump) now produces **valid Oracle SQL that re-imports cleanly**.
The `CREATE TABLE` is generated from the **real source column types** via the
shared table-designer mapping (TASK 46) — `NUMBER`/`NUMBER(p,s)`, `VARCHAR2(n)`,
`CLOB`, `DATE`, `TIMESTAMP`, … never generic `text`/`integer`/`serial` (which
raised ORA-00902) — with `PRIMARY KEY`, `NOT NULL`, and `GENERATED BY DEFAULT AS
IDENTITY` preserved. Because Oracle has **no multi-row `VALUES` list**, data is
emitted as **one `INSERT` per row** there (other engines keep multi-row batching
when enabled). `DATE`/`TIMESTAMP` values are emitted as `TO_TIMESTAMP(...)` so a
bare ISO string doesn't raise ORA-01858 on import. The exporter now reads the
real types for **every** engine's `CREATE TABLE` (the old generic fallback was
shared), so each dialect gets its own valid syntax.

**Views & ER diagram (TASK 61):** Oracle **view create / edit / drop** work
through the shared object editor — `CREATE OR REPLACE VIEW … AS SELECT` (which is
dialect-neutral), the definition round-trips into the editor via `ALL_VIEWS.TEXT`
(and views built with the Visual View Builder save with Oracle's no-`AS` table
aliases). The **ER diagram** now renders Oracle FK relationships — `getTableSpec`
introspects foreign keys from `ALL_CONSTRAINTS` (type `R`) resolved through
`R_CONSTRAINT_NAME` to the parent table/columns — and supports **edit**: draw an
FK (`ALTER TABLE … ADD CONSTRAINT … FOREIGN KEY … REFERENCES …`, with
`ON DELETE CASCADE`/`SET NULL` only — **Oracle has no `ON UPDATE`**, so the FK
dialog never offers it), drop an FK, and create/edit/drop tables via the TASK 46
designer. Dropping a table uses `DROP TABLE … CASCADE CONSTRAINTS PURGE`
(drops referencing FKs and bypasses the recycle bin — noted in the confirm).

Oracle dialect handling: identifiers are UPPERCASE (from `ALL_*` views, quoted
as-is); binds are positional `:1, :2, …`; a **user == a schema**.

**Backlog for Oracle:** materialized views (a distinct object type — not shown in
the Views node), routine debugging, dependency analysis, overload disambiguation,
object/collection types, standalone sequences beyond the current support, and
compound / system / DDL-event triggers and trigger ordering (`FOLLOWS`). The
**cross-engine data-transfer wizard** (table copy between any two engines with a
type-translation matrix) is implemented in TASK 64 — see *Data Transfer* below.

## SQL Server (first-class engine)

Microsoft SQL Server is a selectable engine using **node-mssql** (tedious
backend — pure JS, no native build, and it **packages cleanly** like the other
drivers). Verified against **SQL Server 2022** on Docker (`dbtool-mssql`, port
1433) through the headless smoke and the app UI.

**Authentication** — the connection form has an **Authentication** selector:

- **SQL Server Authentication** (default) — user + password. The portable path;
  works on Windows, macOS, and Docker. This is always available.
- **Windows Authentication** (Integrated) — uses the current Windows account, no
  password. It requires the **native, Windows-only `msnodesqlv8`** component,
  which we **never bundle or auto-install**. It is **detected at connect time**;
  when unavailable the app returns a clear message ("Windows Authentication
  requires the native msnodesqlv8 driver…; use SQL Server Authentication
  instead") and never crashes — mirroring how Oracle Thick mode handles a missing
  Instant Client. **SQL Server Authentication is the portable path.**

The form also exposes **Encrypt (TLS)** and **Trust server certificate** toggles
(both default **on**, since a modern driver defaults `encrypt: true` and a local
Docker server presents a self-signed cert), plus server/port (default **1433**),
database, and an optional named instance.

**Verified basics (SQL Server 2022) — all PASS:** connect (SQL Auth), the
**database → schema (`dbo`) → tables/views** tree, schema-aware catalog +
primary keys, paginated browse (`ORDER BY <pk> OFFSET n ROWS FETCH NEXT m ROWS
ONLY`, deterministic + non-overlapping), autocomplete (real tables + columns via
the CodeMirror MSSQL dialect), and parameterized grid CRUD by PK with **IDENTITY**
(`IDENTITY(1,1)`; the new id comes back via an `OUTPUT INSERTED.*` clause) — a
value with a quote (`O'Brien`) and a **Georgian unicode** string survive intact.
All three filter modes (per-column, funnel nested AND/OR, Custom WHERE) produce
valid MSSQL with **`[bracket]` identifier quoting** and **`@p` named parameters**
(never `$1` / `?` / `:n`), including `LIKE ... ESCAPE` with `%`/`_` escaped, `IN`,
`BETWEEN`, and `IS NULL`. The bottom **Filter SQL panel** renders copy-paste-
runnable MSSQL, bracketing **only where needed** (simple `dbo.customers` stays
bare; reserved/spaced names get `[...]`). SQL Server's default collation is
**case-insensitive**, so `contains` works without the `UPPER()` wrapping Oracle
needs.

**Stage 2 — object management + transfer (all PASS):**

- **Indexes** — the per-table Indexes node lists `sys.indexes` (with column list
  and CLUSTERED/NONCLUSTERED type); create single / multi-column / UNIQUE, edit
  (DROP + CREATE), **rename via `sp_rename … 'INDEX'`**, and drop use the SQL
  Server `DROP INDEX name ON schema.table` form (like MySQL, **not** Oracle).
  PK- and UNIQUE-constraint-backing indexes are **flagged read-only** (drop the
  constraint in the designer instead), detected via `is_primary_key` /
  `is_unique_constraint`.
- **Triggers** — statement-level only: **no `BEFORE`, no `FOR EACH ROW`**. The
  editor offers `AFTER` / `INSTEAD OF`, hides the row/statement selector, and its
  template uses the **`inserted` / `deleted` pseudo-tables** (not NEW/OLD). Create
  / edit apply via **`CREATE OR ALTER TRIGGER`** (SQL Server 2016 SP1+, version-
  detected), and **enable/disable** uses `ENABLE|DISABLE TRIGGER … ON table`.
- **Functions & procedures** — the Functions/Procedures nodes list scalar (`FN`),
  inline-TVF (`IF`), multi-statement-TVF (`TF`) and procedures (`P`) from
  `sys.objects` with an `@param` signature from `sys.parameters`. Templates are
  valid T-SQL (`@params`, `RETURNS`/`AS`/`BEGIN…END`, `CREATE OR ALTER`) and work
  **unmodified**; edit round-trips through `OBJECT_DEFINITION` / `sys.sql_modules`
  and re-applies with `CREATE OR ALTER` (a bare `CREATE` loaded on edit is
  normalized to `CREATE OR ALTER`).
- **Import / export** — CSV / JSON / Excel / SQL all round-trip. Generated SQL
  emits real MSSQL types (`NVARCHAR(n)`/`NVARCHAR(MAX)`, `DECIMAL(p,s)`,
  `DATETIME2`, `BIT`, `INT IDENTITY(1,1)`, …, never generic `text`), wraps
  identity inserts in **`SET IDENTITY_INSERT … ON/OFF`**, caps multi-row `VALUES`
  lists at **1000** row constructors, and uses **`N'…'` unicode literals** so
  Georgian text survives. Table dump **and** whole-database dump both emit **`GO`
  batch separators**, and the restore path's splitter is **GO-aware** (a lone
  `GO` line flushes the batch; a `CREATE PROCEDURE/FUNCTION/TRIGGER` body's many
  `;` are kept in one batch).
- **ER diagram** — renders tables with PK/FK markers and the seeded FK edges
  (`sys.foreign_keys`), and supports **edit** (draw an FK / drop an FK) through
  the SQL Server table-designer DDL (`ALTER TABLE … ADD CONSTRAINT … FOREIGN
  KEY` / `DROP CONSTRAINT`, `sp_rename` for table/column renames).

**`CREATE OR ALTER`** (for routines, triggers, views) needs **SQL Server 2016
SP1+** (major 13+); the driver detects the version. On older servers, in-place
edit of these objects is not offered (drop + create manually).

**Staged for a later SQL Server stage (not yet):** standalone **sequences**
(SQL Server has them, 2012+), extended-property comments, and multi-database
browsing beyond the connected database (the tree scopes to the connection's
database, PostgreSQL-style).

## MariaDB (first-class engine)

MariaDB is selectable in the connection form and backed by a driver that
**extends the MySQL driver** (wire- and SQL-compatible: same `mysql2` connection,
quoting, paging, types, CRUD, DDL, import/export, dump/restore, ER, triggers,
indexes, views, routines) and **overrides only the real divergence — standalone
sequences**. Codegen treats MariaDB as the `mysql` SQL dialect via a central
`sqlDialect()` helper, so every reused MySQL path is correct, while the driver,
object tree, and UI badge keep the real `mariadb` engine.

**All 19 smoke groups PASS on MariaDB 11**, including MariaDB-specific
**sequences** — `CREATE SEQUENCE` / `NEXTVAL()` / `ALTER SEQUENCE` (increment +
`RESTART`) / rename (via `RENAME TABLE`, since MariaDB sequences are tables) /
`DROP SEQUENCE`, plus list + details from `information_schema` and each
sequence's one-row metadata table. The object tree shows a **Sequences** node for
MariaDB (like PostgreSQL).

MariaDB notes:
- **JSON** is a `LONGTEXT` alias (with an implicit `json_valid` CHECK) — stored
  and displayed as text; the seed's JSON columns load and read back fine.
- **CHECK constraints are enforced** (MariaDB 10.2+), unlike MySQL 5.7.
- **No `FULL OUTER JOIN`**, and **no `AS <type>` / `OWNED BY`** on sequences —
  handled by the MySQL-dialect join set and the MariaDB sequence generator.
- `mysql_native_password` auth connects fine via `mysql2`.

   report "no standalone sequences" (empty list, no crash) — handled, not a

**Two real bugs were found by testing and fixed:**

1. **Oracle date/time import → `ORA-01861`.** Importing a CSV/JSON/Excel row whose
   value for a `DATE`/`TIMESTAMP` column is an ISO string (`2024-01-15T09:12:34Z`)
   failed — node-oracledb can't implicitly parse that literal. The Oracle driver
   now converts an ISO date string to a JS `Date` (which node-oracledb binds
   natively) for `DATE`/`TIMESTAMP` columns. This also fixes **grid-editing** of
   Oracle date columns. Oracle's `DATE` keeps its time component through the
   round-trip.
2. **MySQL/MariaDB `CALL procedure()` crashed the driver.** A `CALL` returns an
   *array of result sets* (plus a trailing OK packet); `runQuery` mapped
   `fields → f.name` over the nested/undefined field arrays and threw. It now
   detects a multi-result-set response and uses the first result set (so running
   a stored procedure returns its rows instead of crashing).

**Documented semantic:** on **Oracle** an empty string `''` is stored as `NULL`
(Oracle's rule) — so an imported/transferred `''` reads back as `NULL` there;
every other engine preserves `''`. Oracle `DATE` always carries a time component.

## Data Transfer — any engine → any engine (TASK 64)

Tables can be **copied from any connection to any other**, across all six
engines. The **source is READ-ONLY** — nothing is deleted or modified there; the
selected tables are (re)created on the target with structure + data. Reachable
from the tree via right-click → **Data Transfer…** on a schema (pick tables) or a
table (pre-selected). The wizard shows a per-column **source type → target type**
plan with lossy-mapping **warnings** before running, and offers per-column
override / skip.

**How it works:** structure is recreated by reusing the table-designer DDL
generator (per-engine type rules), then tables are created **without foreign
keys**, all data is copied via **parameterized** batched inserts, and FKs are
added **last** (so integrity is validated once every row is present).
Per-target INSERT mechanics: Oracle **single-row** binds; SQL Server **1000-row
chunks wrapped in `SET IDENTITY_INSERT`** (batch, not `sp_executesql`, so it
persists) to preserve source identity values; the rest **batched multi-row**.

**Type-translation matrix — verified pairs (all PASS on real containers with
disposable targets):**

| ↓ from / → to | PostgreSQL | MySQL | MariaDB | SQLite | Oracle | SQL Server |
| --- | :---: | :---: | :---: | :---: | :---: | :---: |
| PostgreSQL | ✅ | — | — | — | ✅ | ✅ |
| MySQL | — | — | — | — | ✅ | ✅ |
| MariaDB | — | — | — | — | ✅ | ✅ |
| SQLite | — | — | — | — | ✅ | ✅ |
| Oracle | ✅ | ✅ | ✅ | ✅ | — | ✅ |
| SQL Server | ✅ | ✅ | ✅ | ✅ | ✅ | — |

Each pair copied an awkward-data table (auto-increment PK, `O'Brien`, **Georgian
unicode**, empty string, `NULL`, decimal, **datetime with a time component**,
boolean, long text) and verified row counts + values match, identity values are
preserved, warnings appeared, and **the source was unchanged**. A two-table
parent→child set (PG → SQL Server and PG → Oracle) verified **FK-ordered
creation** with the FK recreated on the target.

**Lossy / approximate mappings surfaced as warnings:**

- **Auto-increment** translated `SERIAL ↔ AUTO_INCREMENT ↔ IDENTITY(1,1) ↔ Oracle
  IDENTITY ↔ SQLite AUTOINCREMENT`; **source values are preserved** on copy.
- **Booleans** map across `BOOLEAN / TINYINT / BIT / NUMBER(1)`; `''` → **`NULL`
  on Oracle**; **Oracle `DATE`** → a timestamp/datetime type on other engines so
  the **time component is kept**.
- **`timestamptz`** → the offset is **preserved** on engines that have a
  tz-aware type — SQL Server `DATETIMEOFFSET`, Oracle `TIMESTAMP WITH TIME ZONE`,
  Postgres `timestamptz`; it is dropped (with a warning) only on **MySQL/SQLite**,
  which have no tz type.
- **PostgreSQL arrays** → flattened to text (values as JSON);
  **JSON/JSONB** → `CLOB` (Oracle) / `NVARCHAR(MAX)` (SQL Server) / `TEXT`
  (SQLite) where there is no native JSON; SQL Server **`NVARCHAR(MAX)`** →
  `TEXT`/`CLOB`.
- **BLOB / binary** columns round-trip as bytes (`bytea ↔ VARBINARY(MAX) ↔ BLOB`)
  — the value is decoded back to binary rather than landing as a hex string.
- An **unrecognized** source type is flagged in the plan for review (pick a
  target type or skip the column) — never silently guessed.

**Deep verification (TASK 65)** exercised, beyond the pair matrix, the real
TransferWizard UI (both context-menu entry points, plan preview with visible
per-column mapping + warnings, per-column skip/override honored on the target,
existing-target modes) and these edge cases — all PASS: empty table (0 rows),
composite PK, no PK, **self-referencing** and **cyclic** FKs, multi-table FK
ordering, `DEFAULT`/`NOT NULL` columns, **large volume** (2,500 and 10,000 rows,
crossing the SQL Server 1,000-row chunk boundary; Oracle single-row path),
**100k-char** text → CLOB/`NVARCHAR(MAX)`, **BLOB** bytes, all-`NULL` rows,
identifiers needing quoting (space / mixed-case / reserved word), **append**
(existing rows kept, new added; a PK collision is reported, not silently
partial), and a mid-transfer error (incompatible override → failure reported,
target left in a defined empty state). Two bugs were found and fixed: binary
values were transferred as a hex string (now decoded to bytes), and a NULL in a
SQL Server binary column was mis-bound as `NVARCHAR` (now bound as `VarBinary`).

**Existing target table** options: **Skip**, **Drop & recreate** (confirmed), or
**Append**. A transfer refuses to run when source and target resolve to the same
connection **and** schema (that would write the source) — copy to a different
schema/connection instead.

**Known limitations:** cross-engine **timestamps** move as strings and are
subject to the usual timezone interpretation between engines (the time-of-day is
preserved; on a plain (non-tz) timestamp the wall-clock hour may shift by the
offset). Indexes are recreated best-effort (a name clash is reported, not fatal).
A target sequence/identity counter is **not** advanced past the copied values
(fine for a disposable/target copy; reset it if you then insert new rows without
explicit ids). Very large BLOB values into **Oracle** bind as `RAW` and so are
bounded by Oracle's bind size (small/medium binaries are fine).

## PostgreSQL advanced objects (TASK 67)

PostgreSQL connections show four PG-specific object areas in the tree (hidden for
every other engine). Verified end-to-end against the live PG16 container with
disposable `_pgadv_*` objects; seeds left intact (`customers=20`).

- **Materialized Views** (`pg_matviews`) — a **Materialized Views** tree node
  (separate from Views) showing each matview's populated/unpopulated status.
  Create (SQL editor, `CREATE MATERIALIZED VIEW … AS <select>`), **Refresh data**
  and **Refresh CONCURRENTLY** (needs a UNIQUE index — PostgreSQL errors otherwise,
  which is surfaced), **Open data** (browses it like a table), Drop, and Edit
  (matviews have **no `CREATE OR REPLACE`**, so an edit is DROP + CREATE, confirmed).
- **Types / Enums** (`pg_type` / `pg_enum`) — a **Types** node listing enum +
  composite types with their labels/fields. Create enum + composite (SQL editor),
  `ALTER TYPE … ADD VALUE` (incl. `BEFORE`/`AFTER`), rename type / rename value
  (pre-filled ALTER templates), and Drop (with a CASCADE choice; dropping an in-use
  type is blocked by PG unless CASCADE). User-defined types also appear in the
  **Table Designer's column-type dropdown** (a "User types" group), so an enum can
  be picked as a column type. Invalid enum values are rejected by PostgreSQL and
  the error is surfaced. **Limitation surfaced:** PostgreSQL cannot DROP or reorder
  an enum value — the editor template states this.
- **Extensions** (`pg_extension` / `pg_available_extensions`) — an **Extensions**
  node listing installed extensions, and an **Extensions…** dialog listing all
  available ones with Install / Update / Drop. `CREATE EXTENSION` / `ALTER EXTENSION
  … UPDATE` / `DROP EXTENSION [CASCADE]`. **Caveat surfaced:** some extensions need
  superuser — the server's error is shown rather than crashing.
- **Advanced indexes** — the standalone index editor gains PG-only fields: an
  index **method** (`btree`/`hash`/`gin`/`gist`/`brin`), a **partial** `WHERE`
  predicate, and an **expression** index (e.g. `lower(email)`). The method,
  predicate, and expression are read back from `pg_get_indexdef` + `pg_am` +
  `pg_get_expr(indpred)` so they **round-trip** in the index list and edit form.
  Basic B-tree / UNIQUE / multi-column indexes are unchanged; the other engines'
  index flow is untouched (these fields are PostgreSQL-only). Note: a previous
  bug where **expression indexes were dropped from the index list** (the column
  join on `indkey=0` excluded them) is fixed — they now list correctly.

## Known engine limitations (informational)

These are inherent to the older engine and do **not** break any app feature:

- **MySQL 5.7 — no CTEs (`WITH`) / window functions.** No core app feature
  depends on them: pagination uses `LIMIT/OFFSET`, and the reverse-view parser
  already falls back to raw SQL for constructs it can't model (UNION/INTERSECT/
  EXCEPT). (The smoke suite's own bulk-insert helper was made CTE-free so it
  measures the app, not the helper.)
- **MySQL 5.7 — `CHECK` constraints are parsed but not enforced** (a 5.7
  limitation; 8.0.16+ enforces them). The seed's `CHECK` clauses load without
  error; enforcement simply doesn't happen on 5.7.
- **MySQL — `mysql_native_password`** (5.7 default) vs `caching_sha2_password`
  (8.0 default): both authenticate fine through the `mysql2` driver.
