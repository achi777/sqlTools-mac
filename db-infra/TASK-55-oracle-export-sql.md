# ROLE & CONTEXT
Exporting an ORACLE table to a .sql file produces INVALID SQL that fails on
re-import. Two separate bugs:

BUG 1 — WRONG TYPES (ORA-00902 invalid datatype): the exporter emits generic/
PostgreSQL-SQLite types instead of Oracle types, and appears to ignore the
source column types entirely:
    CREATE TABLE "DBTOOL"."sss" (
      "id" text,      -- Oracle has no "text" type -> ORA-00902
      "name" text
    );
  The real source columns are Oracle types (e.g. NUMBER, VARCHAR2(n)).

BUG 2 — MULTI-ROW INSERT (not yet reached, fails right after Bug 1 is fixed):
    INSERT INTO ... VALUES (1,'a'),(2,'b'),(3,'c');
  Oracle does NOT support multi-row VALUES lists. It needs either separate
  INSERT statements, or INSERT ALL INTO ... VALUES (...) INTO ... VALUES (...)
  SELECT * FROM dual;

Fix the SQL export/dump generator so Oracle output is valid and round-trips
(export -> import works). This is the "Oracle dump/restore dialect" that was
staged out of scope in TASK 29/45.

# FIX 1 — Use the REAL source column types, mapped per dialect
1. The exporter must read the source table's ACTUAL column types (from the
   catalog / getTableSpec — Oracle already returns base type + length/scale +
   IDENTITY from TASK 46) and emit them, not a generic fallback.
2. Emit ORACLE types when the target dialect is Oracle — reuse the Oracle type
   mapping already written for the Table Designer (TASK 46: oracleColumnType).
   Do NOT duplicate a second mapping; share one.
   e.g. NUMBER / NUMBER(p,s), VARCHAR2(n), CLOB, DATE, TIMESTAMP,
   TIMESTAMP WITH TIME ZONE, BLOB, etc. Never emit "text"/"integer"/"serial".
3. Preserve NOT NULL, DEFAULT, PRIMARY KEY, and IDENTITY
   (GENERATED BY DEFAULT AS IDENTITY) as TASK 46 does — the exported CREATE
   TABLE should match what the designer would generate for the same table.
4. AUDIT: check whether this "generic type" fallback also affects export for
   the OTHER engines (does exporting a MySQL/PG table emit correct types?).
   If the exporter has been emitting generic types everywhere and only Oracle
   noticed (because Oracle is strict), fix it for all dialects — each engine's
   export should use that engine's own type syntax.

# FIX 2 — Oracle INSERT syntax
1. When the target dialect is Oracle, do NOT emit multi-row
   "VALUES (...),(...),(...)". Instead emit either:
   (a) one INSERT statement per row (simplest, most portable), or
   (b) INSERT ALL
         INTO t (cols) VALUES (...)
         INTO t (cols) VALUES (...)
       SELECT * FROM dual;   (batched; note INSERT ALL has limits and does not
       work with identity/sequence NEXTVAL in the same way — if unsure, prefer
       (a)).
   Choose one, implement it consistently, and document the choice.
2. Keep multi-row batching for the engines that DO support it (PG/MySQL/
   MariaDB/SQLite) — don't regress them.
3. Statement terminators: ensure the generated Oracle script's statements are
   split correctly on import (the restore path's dialect-aware splitter must
   handle these; PL/SQL blocks end with "/" but plain DML/DDL with ";").
